# RTC + SPI NOR 黑匣子

## 结论

板载 GD25LQ128 容量为 16 MiB，当前 APP A/B 和 UI A/B 分区不需要删除或缩小。原分区表已经在固件 B 槽与 UI A 槽之间预留了 960 KiB 诊断区，当前工程没有其他模块使用该区域，因此黑匣子直接使用：

```text
起始地址：0x00410000
结束地址：0x004FFFFF
区域大小：0x000F0000（960 KiB）
擦除单位：4 KiB
扇区数量：240
```

完整外部 Flash 布局：

| 区域 | 起始地址 | 大小 |
| --- | ---: | ---: |
| Boot 控制与状态 | `0x00000000` | 64 KiB |
| APP A | `0x00010000` | 2 MiB |
| APP B | `0x00210000` | 2 MiB |
| RTC 黑匣子 | `0x00410000` | 960 KiB |
| UI A | `0x00500000` | 5 MiB |
| UI B | `0x00A00000` | 5 MiB |
| 工厂保留 | `0x00F00000` | 1 MiB |

地址契约位于 `Middlewares/ota/ota_layout.h`。黑匣子宏只是诊断区的正式用途别名，没有改变 APP/UI/Boot 的任何地址。

## 存储格式

每个 4 KiB 扇区包含一个 64 字节扇区头和 63 条固定 64 字节记录。总容量为：

```text
240 × 63 = 15120 条记录
```

扇区头保存格式版本、逻辑代际、扇区序号、首记录序号和 CRC32。记录保存：

- 事件类型、严重级别、来源和错误码；
- 从 2000-01-01 开始的 RTC 秒数；
- 本次启动的单调运行毫秒数；
- 最多 28 字节事件负载；
- 记录序号和 CRC32；
- 独立的提交标记。

## 掉电安全规则

扇区头和记录都采用两阶段提交：

1. 先写内容和 CRC32；
2. 读写链路成功后，最后单独写 4 字节提交标记；
3. 启动恢复只接受“魔数、版本、长度、CRC、提交标记”全部正确的对象；
4. 半写记录会计入损坏计数并跳过，不会阻止后续记录恢复；
5. 当前扇区写满后才擦除环中的下一个扇区；新扇区头提交前，旧扇区仍然有效；
6. `clear` 只创建新逻辑代际，不执行 240 个扇区的整区擦除。

这种方式保证单次掉电最多损失正在提交的一条记录，不会因为一个半写对象丢掉整个日志区。

## RTC 策略

- RTC 使用板载 32.768 kHz LSE。
- 不重置备份域，不覆盖已有有效日期。
- 如果备份域从未设置过时间，只在第一次启动时用固件编译时间做近似初始值，并在记录标志中注明 `build_seeded`。
- 后续复位保留 RTC；可通过 shell 设置准确时间。

```text
rtc status
rtc set 2026-07-11 22:30:00
```

## Shell 接口

```text
blackbox status
blackbox tail 8
blackbox mark test_event
blackbox clear confirm
```

`blackbox clear` 要求显式 `confirm`，并且只是开始新代际。普通 RAM 日志中的 warning/error 会由黑匣子后台任务批量镜像；SPI 擦写不发生在产生日志的业务任务或中断中。

## 当前验证证据

- 主机模拟 NOR：空白初始化、跨扇区滚动、重启恢复、半写记录跳过、逻辑清空后恢复全部通过。
- Keil Clean Build：0 Error、0 Warning。
- 真机首次启动：容量 15120、首条 boot 记录提交、CRC 错误 0、SPI I/O 错误 0。
- 真机再次复位：恢复 1 条旧记录并追加第 2 条；没有再次擦除扇区；RTC 保留并继续计时。

尚未宣称通过的项目：真实断电瞬间的示波/电源切断耐久测试、SPI NOR 擦写寿命统计、长期满环压力测试。
