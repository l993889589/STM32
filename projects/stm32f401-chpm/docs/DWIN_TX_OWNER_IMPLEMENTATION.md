# CHPM DWIN 唯一发送入口实现说明

更新日期：2026-07-20  
目标：STM32F401CCU6 + ThreadX  
约束：不修改上位机协议，只编译和测试，不烧录。

## 1. 最终结构

所有业务代码只提交“显示意图”，不再直接操作 DWIN UART：

```text
USB / 传感器 / 报警 / 兼容接口
              |
              v
       dwin_tx_policy.c
  蜂鸣器优先 / 事件FIFO / 最新值覆盖
              |
              v
          dwin_tx.c
  唯一TX线程 + 唯一ACK事务
              |
              v
        drv_dwin_write()
              |
              v
        USART2 -> DWIN
```

`drv_dwin_write()` 在项目源码中只有
`user/dwin/dwin_tx.c` 一个调用点。旧的 `dwin_app.c` 和
`dwin_drv.c` 已改成提交适配器，不能再绕过 owner。

## 2. 三类发送语义

### 2.1 动态数据：latest-wins

温湿度、DS18B20、连接状态、PWM、`0x1100`、`0x1120` 按 VP
地址作为 key 保存。相同 key 尚未发送时，新值直接覆盖旧值。

这符合“动态数据持续刷新，旧值丢失没有意义”的业务条件，也避免慢屏幕
造成旧遥测持续积压。动态帧发送失败后不重试旧值，等待下一次新值刷新。

### 2.2 有序事件：FIFO + 有界重试

页面、颜色、文本、`0x1500/0x1600/0x1700` 和复位请求使用 12
项静态 FIFO。当前业务提交时配置最多重试 2 次；队列满时接口明确返回
`DWIN_TX_STATUS_FULL`，并累计诊断计数。

### 2.3 蜂鸣器：独立锁存 + 无限重试

报警开始后立即向 VP `0x00A0` 发送 `{0x00, 0x3E}`，成功后每
5 秒再次发送。失败时每 100 ms 重试，直到成功或收到明确的停止命令。

蜂鸣器不占普通事件队列，因此事件队列满不会使蜂鸣器任务丢失。每次选取
待发送项时，已到期蜂鸣器优先级最高；重试等待期间允许其他事件使用链路。

## 3. 固定 ACK 处理

DWIN 对不同写请求返回相同固定帧，无法从 ACK 内容反推出对应的 VP。
因此实现严格保持单事务：

1. 发送前打开唯一 request 窗口；
2. 通过 USART2 发送一帧；
3. 只在当前窗口内接受固定 ACK；
4. 同时接受 6 字节无 CRC 和 8 字节带 CRC ACK；
5. 超时后关闭窗口并留 20 ms 排空期，避免迟到 ACK 被下一事务误认。

这解决了多发送者和固定 ACK 错配问题，但没有改变线上的 DWIN 协议。

## 4. 静态资源与诊断

- 事件槽：12 项；
- latest 槽：8 项；
- 最大 DWIN 线帧：258 字节；
- owner 线程栈：2048 字节；
- 不使用动态内存；
- 诊断包括提交、覆盖、拒绝、重试、最终失败、UART 失败、ACK 超时、
  6/8 字节 ACK 成功次数和蜂鸣器成功次数。

## 6. 仍需实机验证的边界

当前完成的是源码、主机测试和目标编译验证。新板到手后仍应验证：

1. 6 字节和 8 字节 ACK 在真实屏幕配置下都能完成事务；
2. 拔掉屏幕或破坏 RX 时，蜂鸣器失败计数持续增加且恢复后自动成功；
3. 高频 USB 动态数据下，同 VP 保持最新值且事件顺序不乱；
4. 报警停止后不再产生新的蜂鸣器命令；
5. ACK 与异步帧紧邻到达时，长度解析器仍能逐帧交付。

第 5 项的源码风险已处理：DWIN RX 现在完全按照 `5A A5 + LEN` 拆帧，
IDLE/20 ms 只丢弃残帧。主机回归已覆盖 ACK 粘包和半包，仍需通过真实
DWIN 屏幕突发帧测试确认电气与时序表现。
