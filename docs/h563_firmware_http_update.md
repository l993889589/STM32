# STM32H563 Firmware HTTP Range Update

## Transport Boundary

MQTT carries only the update command and low-rate status. Firmware bytes are
read from the desktop assistant over HTTP Range and written by the application
through the shared inactive-slot transaction. The W800 task does not publish
normal telemetry while a blocking Range transfer is active; it only sends
bounded MQTT keepalive traffic.

## Desktop Endpoints

The desktop assistant serves firmware on the same HTTP port as UI assets:

- `GET /firmware/manifest.json`
- `GET /firmware/app.bin`
- `Range: bytes=<start>-<end>`
- response header `X-Range-CRC32`

The package consists of `usb_ota_app.bin` and the matching
`usb_ota_app.manifest.json` generated by `make_ota_package.ps1`. The server
rejects a sidecar whose size or CRC differs from the binary.

## MQTT Command

```json
{
  "cmd": "fw_http_update",
  "host": "192.168.1.4",
  "port": 8088,
  "path": "/firmware/app.bin",
  "version": 2026071017,
  "size": 860480,
  "crc32": 127781372,
  "imageFlags": 2,
  "sha256": "29A77359ECD2EEF8BC484900B21624E19309DA3D5A8321AE5F6A50F9752D39BA",
  "signature": "<128 hex characters: raw P-256 r||s>",
  "entryAddress": 134349885,
  "chunkSize": 4096
}
```

The application validates the entry address against internal App flash, chooses
the non-active external firmware slot, and stores only slot-relative offsets.
Each response block must match its Range CRC. After the final block, the shared
transaction rereads the complete external image, validates the image CRC, and
validates SHA-256 before it atomically transitions `VERIFIED -> PENDING`. Boot
re-hashes the selected external slot and verifies the signed descriptor with
the compiled P-256 public key before touching internal App flash.

The desktop assistant also exposes a loopback-only acceptance endpoint:

```text
POST   /__test/http-fault?mode=status&afterOffset=4096&count=1
DELETE /__test/http-fault
```

It is not reachable from LAN clients. `tools/test_desktop_firmware_http.ps1`
uses it to prove that a failed Range does not advance the download and that the
same Range succeeds on retry with an identical CRC.

## USB Protocol

LDOT v2 uses these commands:

| Command | Value | Payload/address |
| --- | ---: | --- |
| Firmware begin | 32 | 124-byte firmware descriptor |
| Firmware data | 33 | slot-relative offset and data |
| Firmware finish | 34 | empty payload |
| Firmware abort | 35 | empty payload |

`flash_app_usb_ota.ps1` uses LDOT v2 by default. `-LegacyV1` is retained only
for the one-time migration from an application that does not understand v2.

## Confirmation Policy

Trial confirmation requires 60 seconds of healthy scheduling from the RS485,
W800 and UI tasks. Network association, MQTT broker reachability and Modbus
slave response are not health requirements. A missing/stale required heartbeat
or ThreadX stack fault prevents confirmation. At 120 seconds without health,
the application resets so the bootloader can count the failed trial and roll
back after the configured limit.
